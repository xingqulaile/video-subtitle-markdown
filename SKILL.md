当用户提供视频文件和 `.srt` 字幕文件时，交付物必须是图文 Markdown 成品，而不是只有纯文本笔记。

优先处理屏幕录制、软件演示、网页操作、代码讲解、产品 walkthrough 这类视频。你的任务不只是整理字幕，还要判断哪些句子需要配图，最后把截图嵌回 Markdown。

## 结果要求

- 最终产物是一个可直接阅读的 `.md` 文件
- Markdown 中既有整理后的文字，也有本地截图图片链接
- 原字幕文字必须完整保留，不得删减、总结或跳写
- 允许补充标点、断句、段落和简短标题

## 输入识别

- 在当前工作目录中寻找 `.srt` 和视频文件
- 视频优先匹配与字幕同名的 `.mp4`、`.mov`、`.mkv`、`.webm`
- 如果存在多个候选文件，优先同名；否则使用第一个，并在最终回复中说明假设

## 工作流

严格按以下顺序执行，不要只做其中一段：

1. 读取 `.srt`，生成 Markdown 初稿
2. 在真正需要视觉辅助的句子末尾插入截图占位符
3. 将 Markdown 保存到当前工作目录
4. 运行 `python3 scripts/replace_screenshots.py`
5. 确认产出最终图文 Markdown，并向用户报告输出路径

## 第一步：字幕转 Markdown

- 使用中文表达，保留必要的英文专有名词
- 第一段作为引言，不加标题
- 其余内容使用单一层级标题，统一写成 `## 标题`
- 按语义整理成 2 到 6 个自然段落，不要碎片化
- 输出 Markdown 正文时不要用代码块包裹
- 必须完整保留字幕所有文字内容，不能压缩

如果一句字幕很短，但和前后句明显属于同一操作流程，可以在保留原文的前提下合并成自然段。

## 第二步：智能插入截图占位符

你的截图目标不是“平均插图”，而是“让读者看见关键操作和关键画面”。

优先给这些句子加截图占位符：

- 演示点击、输入、切换、拖拽、复制、粘贴、提交、运行
- 讲解代码、终端输出、配置文件、报错信息、参数修改
- 提到“这里”“这儿”“这一项”“这个按钮”“这个页面”“可以看到”“如下”这类视觉指代
- 提到网址、页面地址、仓库地址、接口地址、菜单路径
- 对比两个概念、两个界面或前后效果，配图明显更容易理解
- 某一步如果不看画面就难以复现，优先截图

这些情况通常不要截图：

- 寒暄、铺垫、重复总结
- 纯概念说明且不依赖画面
- 连续多句都在描述同一静态画面时，不要密集重复截图

## 占位符格式

- 默认使用 `Screenshot-[mm:ss]`
- 超过一小时的视频可使用 `Screenshot-[hh:mm:ss]`
- 时间取该句对应字幕结束时间，向下取整到秒
- 占位符放在句末，前面保留一句自然语言

示例：

这里点击右上角的 `Settings`。 Screenshot-[02:14]

如果拿不准是否要截图，优先问自己一句：没有这张图，读者会不会更难复现这个操作？如果会，就加。

## 第三步：替换占位符为本地截图

生成 Markdown 初稿后，必须继续运行：

```bash
python3 scripts/replace_screenshots.py
```

脚本职责：

- 自动找到当前目录中的 Markdown 与视频
- 解析 `Screenshot-[mm:ss]` / `Screenshot-[hh:mm:ss]`
- 用 `ffmpeg` 生成截图
- 将占位符替换成 `![screenshot](assets/xxx.jpg)` 形式的本地图片链接
- 输出最终 Markdown 文件路径

## 交付标准

- 如果找到了视频，交付替换完成后的最终 Markdown
- 如果没有视频，保留占位符，并明确告知未生成截图
- 如果 `ffmpeg` 不可用或某个时间点截图失败，保留该占位符，并说明失败原因
